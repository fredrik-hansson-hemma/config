    Trusted CA Bundle Changes

    If you use HTTPS protocol or Secure Socket Layers (SSL) to secure your grid or middle tier, you are using CA certificates to verify the communication process. When your software was initially deployed and a configuration file needed to specify a CA certificate, the SSLCALISTLOC option was used in that file to point to the location of a CA certificate. Unfortunately, every time your software was updated, those configuration files would be overwritten and the SSLCALISTLOC option would have to be added manually again. Beginning with SAS 9.4 (TS1M3), Rev. 940_15w29, the formerly manual process has been automated and the CA certificate bundle moved to a central location so that updating the software will not overwrite any security changes that had been made.

    During an update, the SAS Deployment Wizard reads through configuration files looking for any instance of the manual addition of the -sslcalistloc attribute and comments it out. This was done in order that the attributes would not interfere with the new trusted CA bundle provided with your updated software. After the update, your software will use the new trusted CA bundle unless you return to the file and remove the comments. The following is a list of files that have had that attribute commented out so that you can open the file manually and remove the comments in order that the attribute can still be used if you'd like.

    Any self-signed or site-signed CA certificates that have previously been added to the SAS Private JRE's cacerts file manually must also be added to the new trusted CA bundle using the SAS Deployment Manager's Add Certificate task, described in the SAS Deployment Wizard 9.4 and SAS Deployment Manager 9.4: User's Guide. Adding the CA certificate should be repeated for each machine in the deployment.

    SAS Visual Analytics Sample Reports

    To help you explore the reporting capabilities of your SAS Visual Analytics environment, sample reports and sample tables are included in your deployment. To ensure that all users can view the samples, an access control that denied the ReadMetadata permission to the PUBLIC group on the /Products/SAS Visual Analytics metadata folder has been removed. If you want to review or adjust access to that folder:

    1. As a SAS Administrator, log on to SAS Management Console.
    2. On the Folders tab, navigate to /Products/SAS Visual Analytics.
    3. Right-click on the folder and select Properties. Then select the Authorization tab.
    4. Note: If you want to reinstate the access control that was removed, select the PUBLIC group and then click the Deny check box for the ReadMetadata permission. (The check box background will change from grey to white.)
    5. Click OK.

    For more information about Sample Reports, see the topic "Sample Objects" in the SAS Visual Analytics: Administration Guide

    SAS Web Application Server

    Backup for your SAS Web Application Server contents

    Due to a major upgrade to the SAS Web Application Server, when you update your software to 9.4M3 or later, the existing SAS Web Application Server information at /opt/sas/config/Lev1/Web/WebAppServer/[server_name] will be backed up before new contents are created. The backup location is:

    /opt/sas/config/Lev1/Web/WebAppServer/[server_name]/Backup/[server_name].[timestamp]

    where [server_name] usually is in the form of SASServerM_N and [timestamp] is the time you run the SAS Deployment Wizard.

    This approach ensures a running system after an update. We expect most customers will not need to do anything after an update. However, if customers have customized the contents in the server instances, they should consult the backup folder and re-apply the customization to the newly created server instances.
