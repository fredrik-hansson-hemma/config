Warnings and Notices

The following issues occurred during the automated portion of your configuration and must be addressed before proceeding.

  • Some of the clients in your environment may need access to the SAS/GRAPH Java applets to view graphical output generated by SAS processes. If so, add the APPLETLOC option to the sasv9_usermods.cfg file located in each SAS Application Server context directory specifying the location to be used in generating graphical output. APPLETLOC should point to a network accessible copy of the SAS/GRAPH Java applets.

    Since your deployment includes the SAS Web Infrastructure Platform, the applets can be made available via:

    -APPLETLOC="http://web-server:port/sasweb/graph"

    The web server port is typically 80 on Windows deployments and 7980 on other deployments. Deployments that do not use a web server can point directly to the SAS Web Application Server, which typically uses port 8080.
    NOTE: Do not add this line to the sasv9_usermods.cfg file until the SAS middle tier is configured. After updating the file, restart the object spawner and validate your Workspace, Pooled Workspace, and the Stored Process servers again.

  • SAS Web Applications are being protected by a security filter that prevents linking from unauthorized hosts. The security filter is customizable. For further information on customizing this filter, refer to the SAS 9.4 Intelligence Platform: Middle-Tier Administration Guide, found at http://support.sas.com/documentation/onlinedoc/intellplatform/index.html .

  • CAUTION: Guest access is enabled for SAS Visual Analytics Hub, so it is essential to limit access for the SAS Anonymous Web User. See the SAS Intelligence Platform: Middle-Tier Administration Guide at http://support.sas.com/documentation/onlinedoc/intellplatform/tabs/admin94.html.

  • Permissions on the logs directory (/opt/sas/config/Lev1/Web/Logs) are originally set at 777 but may change based on the installer ID. We strongly recommend that you narrow access to this directory. To facilitate creation of logs, make sure that the SAS Installer (sas) and the invoker of the web application server can write to this directory. To facilitate appropriate use of the logs, make sure that administrators can see the directory.

Notes:

  • The /opt/sas/config/Lev1/SASApp_VA/WorkspaceServer/Logs directory is intended to serve as a placeholder for workspace server logs. Workspace server logs may be directed to this directory in the event that your SAS or IT administrator enables logging. The following default attributes are associated with the Logs directory:
    • Universal read, write or execute access is disabled for the directory in order to comply with default installation security requirements
    • Execution of a workspace server does not populate the Logs directory
    • The Logs directory serves as a placeholder should you enable logging for workspace servers

    Enabling workspace server logging should be carefully considered, due to the proliferation of workspace server logs, as well as the requirement to grant universal write access to the Logs directory. To enable workspace server logging, perform the following steps:
    • Enable read, write and execute access to the Logs directory
    • Modify the workspace server logconfig.xml to enable the desired levels of logging
    • Alternatively, enable the logconfig.trace.xml logging file to capture workspace server diagnostic logging

PostgreSQL Configuration Manual Steps

Use the following steps to manually configure PostgreSQL inventory configuration properties in the SAS Environment Manager system.

1. Log on to the SAS Environment Manager Web Console.

2. Go to the monitor page of PostgreSQL server by clicking server name of PostgreSQL on the resources list page.

3. Go to the configuration page by clicking the Configure Server button in the Tool Menu.

4. Supply values for the following properties.

postgresql.host Host name of the PostgreSQL server.
postgresql.port Listening port of the PostgreSQL server.
postgresql.user User name of the PostgreSQL server.
postgresql.pass Password for the specified user.
postgresql.pgdata Full path to the data file.
postgresql.program Full path of the postgres.sh command file.

	httpdctl stop

Log Files

There are potentially three log files generated by the server and when any of them reaches a size of 50 megabytes, it is saved and a new log file of that type is started. Log files are placed in the directory /opt/sas/config/Lev1/Web/WebServer/logs. Each log file includes a datestamp as part of the file name. These logs are not removed and should be periodically cleaned up to recover disk space. The following list describes the three different log files that can be generated:
  • access_YYYY_MM_DD-HH-MM.log - plain text http requests
  • error_YYYY_MM_DD-HH-MM.log - server error log
  • ssl_request_YYYY_MM_DD-HH-MM.log - secure requests, (only when SSL is configured)

SAS Deployment Backup and Recovery Tool

Log directory
  • /opt/sas/config/Lev1/Web/Logs/SASServer1_1/SASDeploymentBackup9.4.log
Configuration
  • Because a central vault location was not specified, the feature to take backups at a central location has been disabled.
  • Backup data will be retained for 30 days and purged automatically after this period.
  • Backup is scheduled to run each Sunday at 1:00 a.m. local time.
  • Backup will be taken of the following sources: SAS Metadata Server, SAS Content Server, SAS Web Infrastructure Platform database, Data directories, SASEnvironment directories, and server configuration directories for servers on the SAS server tier.
Validation steps
  1. Log on to SAS Environment Manager as a user that has sasadm@saspw as a username.
  2. Click Administration link in order to open the SAS Environment Manager Administration in a separate window.
  3. In the Side menu of the SAS Environment Manager Administration, select SAS Backup Manager. The SAS Backup Manager appears.

    Note: The SAS Backup Manager might take several minutes to appear as it discovers the assets that are available for backup in the deployment.

  4. In the SAS Backup Manager, select Schedule from the list, and then verify the default schedule

Notes:

  • To register additional directories for backup you need to run the rest utility "sas-update-backup-config" by passing your directory path as value of "sourcetype" parameter (i.e. sas-update-backup-config -sourcetype customdir).
  • To use the SAS Deployment Backup tool in a heterogeneous environment where content server is clustered, it is required that you enable backup at a central vault location. Follow the steps in "Specifying a Central Vault Location in a Heterogeneous Environment" in the SAS Intelligence Platform: System Administration Guide at http://support.sas.com/94administration(A heterogeneous environment is one in which the host machines that are included in the backup are in different operating system families.)
  • For more information about the SAS Deployment Backup Tool, see the SAS Intelligence Platform: System Administration Guide at http://support.sas.com/94administration

Backing Up Your SAS Deployment

The SAS system now includes a Deployment Backup utility that can back up the following components of your SAS deployment:

  • the metadata server
  • selected data and configuration directories on any SAS server tier
  • the SAS Content Server repository
  • any SAS Web Infrastructure Data Server databases
  • custom directories configured for your site
If the Deployment Backup utility is configured on your system, it will automatically run every Sunday at 1 a.m. The automatic backup schedule can be changed or disabled. For more information, see "Using the Deployment Backup and Recovery Tool" in the SAS Intelligence Platform: System Administration Guide at http://support.sas.com/94administration.

Caution: Regular backups of your SAS system are crucial to ensuring the health of your SAS 9.4 Intelligence Platform configuration. It is recommended that you schedule regular system backups to occur just after the SAS metadata or deployment backups so that each system backup includes a timely backup of your SAS components. For more information, see "Best Practices for Backing Up and Restoring Your SAS Content" in the SAS Intelligence Platform: System Administration Guide at http://support.sas.com/94administration.
